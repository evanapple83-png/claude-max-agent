# Technical writing

- **Know the reader and their goal.** Write to what they're trying to accomplish, at their level. A tutorial, a reference, and a concept doc are different shapes — pick the right one.
- **Lead with the outcome.** Say what the reader will be able to do (or what the thing is) before the detail. Don't bury the point under preamble.
- **One idea per paragraph; steps for procedures.** Numbered steps for "how", prose for "why", tables for structured comparisons.
- **Be concrete.** Real examples, real commands, real output. Show, then explain. Examples should be runnable/copy-pasteable.
- **Plain language.** Short sentences. Define jargon on first use or avoid it. Active voice. Cut hedging and filler.
- **Make it scannable** — descriptive headings, short paragraphs, lists, and code blocks so readers can find their spot.
- **Accuracy over polish.** Verify every command, claim, and version. Docs that are wrong destroy trust faster than docs that are plain.
- **Anticipate failure.** Note prerequisites up front, and include the common errors and how to fix them.
